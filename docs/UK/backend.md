## Сутність Carrier

Сутність [Carrier](./../../app/Domain/Carrier/Carrier.php) представляє собою перевізника, із усіма його можливими атрибутами і правилами. Це найголовніша частина блоку. Саме тут знаходяться усі правила та логіка.

Файли розташовані в каталозі [Domain](./../../app/Domain/Carrier/).

Валідація відбувається всередині VO, які представляють атрибути `Carrier`. Усі слої застосунку мають використовувати VO для передачі інформації від одного шару до іншого. Це гарантує правильність вхідних даних та їх незмінність під час обробки.

Підрахунок відбувається в методі `calculateShippingPriceByWeight`. Для підрахунку використовується поле `$calculator` з типом даних `ShippingCostCalculatorInterface`. Таким чином реалізація підрахунку відокремлюється від класу `Carrier`. Реалізація калькуляторів знаходиться в шарі [Application](./../../app/Application/CarrierService/ShippingCostCalculators/). Але це не обов'язково. Ці файли можна (і можливо варто) перенести в Domain.

## Сервіс CarrierService

За роботу з сутністю `Carrier` відповідає сервіс [CarrierService](./../../app/Application/CarrierService/CarrierService.php). Він знає як створити, змінити та знайти потрібного Carrier.

Для роботи з запитами до неї, сервіс використовує команди. Кожна точка входу (HTTP, CLI та інші) має використовувати команди для взаємодії з сервісом.

Команда - це клас, який приймає дані, перевіряє їх і приводить в такий стан, щоб сервіс або сутніть могли ними користуватися. Для зменшення можливих варіантів, дані приводяться до рядка. Приклад команди - [CalculateCommand](./../../app/Application/CarrierService/CalculateShippingCosts/CalculateCommand.php). Вона приймає дані або через конструктор, або через метод `fromHash`. У першому випадку користувач (контролер) має привести вхідні дані до строки (інакше буде помилка при створенні команди). В другому випадку користувач, не обробляючи вхідні дані взагалі, передає їх у вигляді масиву (який є по суті словником).

Команди є першим етапом валідації даних. Усі дані приводяться до рядка і вже потім потрапляють в VO сутностей де проводиться подальша перевірка і приведення до відповідного типу.

Усі користувачи служби мають використовувати команди для створення своїх структур даних. Прикладом може бути назва поля [html форми](./../../resources/views/calculate.blade.php), ім'я якої записано в [команді](./../../app/Application/CarrierService/CalculateShippingCosts/CalculateCommand.php) як `public const slugField = 'carrier'`.

```blade
<label for="{{ $carrier_slug_field }}">Carrier:</label>
```

## HTTP-контролер

[HTTP-контролер](./../../app/Http/Controllers/CarrierCalculateFormController.php) оброблює дві точки входу:  
  
- Відображення HTML форми
- POST API отримання даних з форми і видачи результату підрахунку

За відображення форми відповідає метод `index`. Тут формується список перевізників для відображення. Кожен перевізник представлений у вигляді [DTO](./../../app/Application/CarrierService/List/ListDto.php).

За обробку надісланих даних відповідає метод `calculate`, до завдань якого входить:

- отримати дані з запиту,
- створити команду `CalculateCommand`,
- передати команду в службу `CarrierService`,
- повернути отриманий від служби результат клієнту.

Контролер не працює напряму з результатами, а передає отриману від служби `CalculateView` DTO у відповідну view-DTO, де вже відбувається робота з даними.
Це `SuccessDto` або `ErrorDto` залежно від успіху обробки запиту.

## Репозиторій Carrier

Клас [CarrierRepositoryUsesBuilder](./../../app/Infrastructure/Persist/Carrier/CarrierRepositoryUsesBuilder.php) відповідає за створення, пошук та зберігання сутності `Carrier`. Для взаємодії зі сховищем використовується [Query Builder](https://laravel.com/docs/12.x/queries).

Сервіс `CarrierService` взаємодіє через інтерфейс [CarrierRepositoryInterface](./../../app/Application/CarrierService/RepositoryException.php) та використовує виключення [NoSuchCarrierException](./../../app/Application/CarrierService/NoSuchCarrierException.php) та [RepositoryException](./../../app/Application/CarrierService/RepositoryException.php). Завдяки цьому сервіс відокремлюється від конкретної реалізації репозиторію.
