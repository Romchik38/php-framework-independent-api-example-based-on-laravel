## Опис

Застосунок складається з одного єдиного блоку - підрахунок вартості перевезення в залежності від ваги посилки.

## Сутність Carrier

Сутність [Carrier](./../../app/Domain/Carrier/Carrier.php) представляє собою перевізника, зі всіма його можливими атрибутами і правилами. Це найголовніша частина блоку. Саме тут знаходяться усі правила і логіка.

Файли розташовані в каталозі [Domain](./../../app/Domain/Carrier/).

Валідація відбувається в середині VO, які представляють атрібути `Carrier`. Усі слої застосунку мають використовувати VO для передачи інформації від одного шару до іншого. Це гарантує правильність вхідних даних та іх незмінність під час обробки.

Підріхунок відбувається в методі `calculateShippingPriceByWeight`. Для підрахунку використовується поле `$calculator` з типом даних `ShippingCostCalculatorInterface`. Таким чином реалізація підрахунку відокремлюється від класу `Carrier`. Реалізація калькуляторів знаходиться в шарі [Application](./../../app/Application/CarrierService/ShippingCostCalculators/). Але це не обов'язково. Ці файли можна(і можливо варто) перенести в Domain.

## Служба CarrierService

За роботу з сутність `Carrier` відповідає служба [CarrierService](./../../app/Application/CarrierService/CarrierService.php). Вона знає як створити, змінити та знайти потрібного Carrier.

Для роботи з запитами до неї, служба використовує команди. Кожна точка входу (Http, cli та інші) мають використовувати команди для користування сервісом.

Команда - це клас, який приймає дані, перевіряє їх і приводить в такий стан, щоб служба або сутніть могли ними користуватися. Для зменьшення можливих варіантів, дані приводяться до строки. Приклад команди - [CalculateCommand](./../../app/Application/CarrierService/CalculateShippingCosts/CalculateCommand.php). Вона приймає дані або через конструктор, або через метод `fromHash`. В першому випадку користувач (контролер) має привести вхідні дані до строки (інакше буде помилка при створенні команди). В другому випадку користувач, не обробляючи вхідні дані завсім, передає їх у вигляді масиву (який є по суті словником).

Команди є першим етапом валідації даних. Усі дані приводіться до строки і вже потім потрапляють в VO сутностей де проводиться подальша перевірка і приведення до відповідного типу.

Усі користувачи служби мають використовувати команди для створення своїх структур даних. Прикладом може бути назва поля [html форми](./../../resources/views/calculate.blade.php), ім'я якої записано в команді як `public const slugField = 'carrier'`.

```blade
<label for="{{ $carrier_slug_field }}">Carrier:</label>
```

## Http контролер

[Http контролер](./../../app/Http/Controllers/CarrierCalculateFormController.php) оброблює дві точки входу:  
  
- Відображення html форми
- Post Api отримання даних з форми и видачи результату підрахунку

За відображення форми відповідає метод `index`. Тут формується список перевізників для відображеня. Кожен перевізник представлен у вигляді [DTO](./../../app/Application/CarrierService/List/ListDto.php).

За обробку надісланих даних відповідає метод `calculate`, в завдання якого входить:

- отримати дані з запиту,
- створити команду `CalculateCommand`,
- передати команду в службу `CarrierService`,
- повернути отриманий від служби результат клієнту.

Контролер не працює напряму з результатами, а передає отриману від служби `CalculateView` DTO в одну зі своїх view DTO, де вже відбувається робота з даними.
Це `SuccessDto` або `ErrorDto` в залежності від успіху обробки запиту.

## Репозиторій Carrier

Класс [CarrierRepositoryUsesBuilder](./../../app/Infrastructure/Persist/Carrier/CarrierRepositoryUsesBuilder.php) відповідає за створення, пошук та зберігання сутності `Carrier`. Для зв'язку зі сховищем використовується [Query Builder](https://laravel.com/docs/12.x/queries).

Служба `CarrierService` зобов'язує репозиторій використовувати свій інтерфейс [CarrierRepositoryInterface](./../../app/Application/CarrierService/RepositoryException.php) та помилки [NoSuchCarrierException](./../../app/Application/CarrierService/NoSuchCarrierException.php) та [RepositoryException](./../../app/Application/CarrierService/RepositoryException.php). Завдяки цьому служба відокремлюється від конкретної реалізації репозиторія.
